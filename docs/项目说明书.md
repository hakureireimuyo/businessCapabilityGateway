# 业务能力网关（Business Capability Gateway）— 项目说明书

## 1. 项目概述

### 1.1 项目定位

**业务能力网关**是一个面向 AI Agent 的轻量级协议网关。它不执行任何业务逻辑，只负责：

- 插件的发现与加载
- 请求的解析与路由
- 管道的构建与执行
- 结果的标准化返回

所有复杂的业务能力全部封装在插件内部，网关本身对业务一无所知。

### 1.2 核心哲学

```
网关负责协议
插件负责业务
数据库负责存储
```

**Agent 永远不能接触原始数据。** 所有数据查询、过滤、分析、评分均在插件内部的内存管道中完成，Agent 只能拿到最终的业务结果。

### 1.3 项目命名

| 层级 | 名称 | 说明 |
|------|------|------|
| 项目名 | Business Capability Gateway | 业务能力网关 |
| 协议层 | BCL（Business Command Language） | 业务指令语言 |
| 执行单元 | Node | 插件注册的业务节点 |

## 3. 核心概念

### 3.1 Node（节点）

Node 是插件暴露给网关的**唯一抽象单元**。网关只认识 Node，不认识 Repository、Service、Analyzer 等插件内部概念。

#### 节点类型

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│ SourceNode  │ ──▶ │TransformNode │ ──▶ │  SinkNode   │
│  数据源节点  │     │  转换节点     │     │  输出节点    │
└─────────────┘     └──────────────┘     └─────────────┘
  产生数据集合         修改数据集合         输出最终结果

  例如：              例如：              例如：
  关键词搜索           数据过滤            市场分析
  产品搜索            数据排序            竞争分析
  分类搜索            数据聚合            机会评分
                      数据去重            销量预测
```

#### 管道合法性规则

```
合法：
  Source → Transform → Transform → Sink  ✅
  Source → Sink                          ✅

非法：
  Sink → Transform                       ❌ （Sink 已是终点）
  Transform（无前置 Source）              ❌ （无数据来源）
  Source → Source                        ❌ （无意义串联）
```

### 3.2 Pipeline（管道）

管道是 Node 的有序组合，在一次请求中完整执行，**不产生任何中间状态的持久化**。

```
一次请求 = 一个完整管道 = 一次完整执行
```

执行模型：

```python
context = None
for node in pipeline:
    context = node.execute(context)
return context.result  # 仅当最后节点为 SinkNode 时有值
```

### 3.3 BCL（Business Command Language）

业务指令语言，是 Agent 与网关之间的**唯一协议**。

#### 语法格式

```text
<plugin>/<node_name>&<param1>=<value1>&<param2>=<value2>
> <node_name>&<param>=<value>
> <node_name>()
```

#### 示例

```text
amazon/关键词搜索&keyword=halloween garland
> 数据过滤&review<100&price_gte=20
> 市场分析()
```

#### 设计约束

- **无变量**：不支持 `let x = ...`
- **无控制流**：不支持 `if`、`for`、`while`
- **无状态**：每次请求都是完整独立的指令
- **管道符 `>`**：表示数据从前一节点流向下一节点

### 3.4 ExecutionContext（执行上下文）

管道中节点间传递的唯一数据载体：

```python
class ExecutionContext:
    request_id: str           # 请求唯一标识
    plugin_name: str          # 当前插件名
    pipeline: list            # 当前管道定义
    data: Any                 # 当前数据集（仅内存中存在）
    metadata: dict            # 元数据（耗时、行数等）
```

**ExecutionContext 永远不出网关。** Agent 永远拿不到 ExecutionContext。

#### 成功

```json

    "success": true,
    "data": { ... },
    "message": ""
}
```

#### 失败

```json
{
    "success": false,
    "error": {
        "code": "NODE_NOT_FOUND",
        "message": "节点 '数据过滤' 在插件 'amazon' 中未找到"
    }

```

### 5.4 错误码规范

| 错误码 | 说明 |
|--------|------|
| `INVALID_DSL` | DSL 语法错误 |
| `PLUGIN_NOT_FOUND` | 插件不存在 |
| `NODE_NOT_FOUND` | 节点不存在 |
| `INVALID_PARAMS` | 参数校验失败 |
| `INVALID_PIPELINE` | 管道结构非法（如 Sink → Transform） |
| `RESULT_TOO_LARGE` | 返回结果超大 |
| `EXECUTION_TIMEOUT` | 执行超时 |
| `INTERNAL_ERROR` | 内部错误 |

## 6. Node 开发规范

### 6.1 Node 基类

```python
from abc import ABC, abstractmethod
from typing import Any
from core.execution_context import ExecutionContext


class NodeType:
    SOURCE = "source"
    TRANSFORM = "transform"
    SINK = "sink"


class Node(ABC):
    """插件暴露给网关的唯一抽象"""

    name: str              # 业务名称（如 "关键词搜索"）
    description: str       # 业务说明
    node_type: str         # source | transform | sink
    parameters: dict       # 参数 schema
    input_schema: dict     # 输入类型定义（可选）
    output_schema: dict    # 输出 schema

    @abstractmethod
    def execute(self, context: ExecutionContext) -> ExecutionContext:
        """执行节点逻辑，返回更新后的上下文"""
        ...

    def validate_params(self, params: dict) -> bool:
        """参数校验"""
        ...
```

### 6.2 插件入口

每个插件通过 `plugin.py` 向网关注册其所有 Node：

```python
# plugins/amazon/plugin.py
from core.node_registry import register_nodes
from plugins.amazon.nodes.source_nodes import KeywordSearchNode
from plugins.amazon.nodes.transform_nodes import FilterNode
from plugins.amazon.nodes.sink_nodes import MarketAnalysisNode


def register():
    register_nodes([
        KeywordSearchNode(),
        FilterNode(),
        MarketAnalysisNode(),
    ])
```

### 6.3 Node 实现示例

```python
# plugins/amazon/nodes/source_nodes.py
class KeywordSearchNode(Node):
    name = "关键词搜索"
    description = "根据关键词搜索相关产品"
    node_type = NodeType.SOURCE
    parameters = {
        "keyword": {"type": "string", "required": True},
        "limit": {"type": "integer", "required": False, "default": 100},
    }
    output_schema = {"type": "ProductCollection"}

    def execute(self, context: ExecutionContext) -> ExecutionContext:
        keyword = context.current_params["keyword"]
        products = self.service.search_by_keyword(keyword)
        context.data = products
        return context
```

```python
# plugins/amazon/nodes/transform_nodes.py
class FilterNode(Node):
    name = "数据过滤"
    description = "按条件过滤产品集合"
    node_type = NodeType.TRANSFORM
    parameters = {
        "review_lt": {"type": "integer", "required": False},
        "price_gte": {"type": "float", "required": False},
    }
    input_schema = {"type": "ProductCollection"}
    output_schema = {"type": "ProductCollection"}

    def execute(self, context: ExecutionContext) -> ExecutionContext:
        products = context.data
        params = context.current_params
        if "review_lt" in params:
            products = products.filter_review_lt(params["review_lt"])
        if "price_gte" in params:
            products = products.filter_price_gte(params["price_gte"])
        context.data = products
        return context
```

```python
# plugins/amazon/nodes/sink_nodes.py
class MarketAnalysisNode(Node):
    name = "市场分析"
    description = "分析市场竞争情况并返回评分"
    node_type = NodeType.SINK
    parameters = {}
    input_schema = {"type": "ProductCollection"}
    output_schema = {
        "market_size": "integer",
        "avg_price": "float",
        "competition_score": "integer",
    }

    def execute(self, context: ExecutionContext) -> ExecutionContext:
        products = context.data
        result = self.analyzer.calculate(products)
        context.result = result  # SinkNode 设置最终结果
        return context
```

## 7. 插件内部规范

### 7.1 分层规范

插件内部必须遵循三层架构：

```
Node（节点层）
  ↓  负责：参数接收、结果返回
Service（业务层）
  ↓  负责：业务逻辑、统计分析、评分算法
Repository（数据层）
  ↓  负责：数据库查询、数据获取
Database（数据库）
```

### 7.2 约束清单

| # | 规则 | 说明 |
|---|------|------|
| 1 | Node 是唯一暴露单元 | 不暴露 Repository、Service 给网关 |
| 2 | Node 必须无状态 | 每次 execute 独立，不共享可变状态 |
| 3 | 统一请求格式 | 永远 `plugin/node&params` |
| 4 | 统一返回格式 | 永远 `{success, data/error, message}` |
| 5 | 能力自发现 | Agent 通过 `/plugins` 探索，不依赖 Prompt |
| 6 | 声明输入输出 | 每个 Node 必须有 input_schema / output_schema |
| 7 | 禁止原始数据输出 | 不返回产品列表、SQL结果、表行数据 |
| 8 | Repository 禁止业务逻辑 | 数据访问层只做 CRUD |
| 9 | 统一结果对象 | 使用 ActionResult 而非裸 dict |
| 10 | 超大结果保护 | 超过 MAX_RESULT_SIZE 返回错误 |
| 11 | 执行上下文传递 | 使用 ExecutionContext 贯穿整个调用链 |
| 12 | 插件自包含 | 插件之间不互相调用 |
| 13 | 业务问题放插件 | 永远不把业务逻辑写进网关 |
| 14 | 新增插件不改网关 | Open for Extension, Closed for Modification |

### 7.3 管道数据流

```
数据库
  ↓
Repository.filter().filter().fetch()
  ↓
ProductCollection（内存对象）
  ↓
Service.analyze()
  ↓
Result（dict）
  ↓
返回 Agent
```

**整个过程全部在插件内部的内存中完成，不产生 CSV、JSON 文件、临时表等中间产物。**

## 8. DSL 解析器设计

### 8.1 解析流程

```
输入字符串:
  "amazon/关键词搜索&keyword=halloween&limit=50
   > 数据过滤&review_lt=100
   > 市场分析()"

      ▼
┌─────────────┐
│   Lexer     │  词法分析：拆分为 Token 流
└──────┬──────┘
       ▼
┌─────────────┐
│   Parser    │  语法分析：构建 AST（管道节点树）
└──────┬──────┘
       ▼
┌─────────────┐
│  Validator  │  语义校验：节点类型链合法性、参数校验
└──────┬──────┘
       ▼
┌─────────────┐
│  Pipeline   │  构建可执行管道对象
└─────────────┘
```

### 8.2 AST 结构

```python
# 解析结果
Pipeline(
    plugin="amazon",
    nodes=[
        PipelineNode(name="关键词搜索", params={"keyword": "halloween", "limit": "50"}),
        PipelineNode(name="数据过滤", params={"review_lt": "100"}),
        PipelineNode(name="市场分析", params={}),
    ]
)
```

## 10. 扩展方式

### 新增数据库

```
plugins/
└── steam/          # 新建目录，实现 Node、Service、Repository
    ├── nodes/
    ├── services/
    ├── repository/
    └── plugin.py
```

### 新增业务能力

在已有插件中新增一个 Node 类即可：

```python
class PlayerRetentionNode(SinkNode):
    name = "玩家留存分析"
    ...
```

### 网关代码修改量

**0 行。** 这就是 "Open for Extension, Closed for Modification" 原则的体现。

## 12. 与其他架构的对比

| 维度 | 传统 API Gateway | 分析引擎 | **业务能力网关（本项目）** |
|------|-----------------|---------|--------------------------|
| 抽象单元 | Endpoint | Metric/Dataset | **Node** |
| 业务逻辑位置 | 微服务内部 | 引擎内部 | **插件内部** |
| Agent 可见内容 | API 文档 | 数据模型 | **业务能力列表** |
| 数据暴露 | 可能暴露原始数据 | 可能暴露原始数据 | **仅暴露业务结果** |
| 扩展方式 | 新增服务 | 新增分析模型 | **新增插件（不改网关）** |
| 状态管理 | 可能有 Session | 可能有状态 | **完全无状态** |
