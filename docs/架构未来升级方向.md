
# Business Capability Gateway 架构演进规划（Future Upgrade Plan）

## 一、当前状态（v2.0）

当前系统处于**单人使用阶段的轻量架构**，核心特征如下：

### 1. 插件形态

* 插件以 Python module 形式存在
* 通过继承 Node 基类实现统一接口
* 与 Gateway 强耦合运行（in-process execution）

### 2. Gateway 职责

* 插件加载与注册
* Python sandbox 执行
* Node 调度与校验
* Graph 执行与结果收集

### 3. Agent 职责

* 自然语言理解
* tag 生成与 alias 扩展
* Graph 编排与决策
* 最终执行策略选择

### 4. 系统约束

* 单进程运行
* 无分布式需求
* 无插件隔离需求
* 以开发效率优先

---

## 二、当前架构隐含问题（仅记录，不作为当前约束）

随着插件复杂度增加，当前继承式插件体系可能面临以下结构性限制：

### 1. 插件耦合问题

* 插件必须依赖 Gateway 内部 BaseNode
* 插件生命周期绑定主系统
* 插件无法独立演进

### 2. 扩展性限制

* 无法跨语言实现插件
* 无法独立部署或隔离执行
* 无法形成真正独立能力服务

### 3. 类型系统局限

* 统一性依赖继承结构
* 强耦合 Python runtime
* schema 不具备外部可验证性

---

## 三、未来架构升级方向（长期规划）

> 本章节为架构演进备选方案，仅用于未来扩展参考，当前不强制实施。

---

# 四、升级目标

将系统从：

> **单体插件执行系统（monolithic plugin runtime）**

升级为：

> **基于契约的分布式能力执行系统（contract-driven capability network）**

---

# 五、核心演进方向

## 5.1 插件形态演进

### 当前模式（继承式插件）

```text
BaseNode → PluginNode → Gateway执行
```

---

### 未来模式（独立服务插件）

```text
Plugin Service (any language)
        ↑
   Contract Definition
        ↑
     Gateway RPC
```

---

### 变化本质

* ❌ 不再依赖继承
* ✔ 依赖声明式契约（schema）

---

## 5.2 类型系统升级（从继承到契约）

### 当前

* Python inheritance
* runtime type check

---

### 未来

采用三层契约系统：

#### 1. 输入契约（Input Schema）

定义节点输入结构

#### 2. 输出契约（Output Schema）

定义节点输出结构

#### 3. 行为契约（Behavior Contract）

定义节点语义约束：

* 是否无状态
* 是否确定性
* 排序规则
* 批处理规则

---

## 5.3 Gateway 演进

### 当前 Gateway

* 插件加载器
* Graph 执行器
* Python sandbox runtime

---

### 未来 Gateway

转变为：

> **Contract-aware execution router**

职责变为：

* schema registry
* contract validation
* RPC routing
* execution orchestration
* sandbox isolation（仅针对 Agent script）

---

## 5.4 插件运行模型演进

### 当前（in-process）

* plugin = Python module
* 同进程执行
* 共享 runtime

---

### 未来（out-of-process）

* plugin = independent service
* HTTP / RPC 调用
* 完全隔离运行时

---

## 5.5 Agent 层变化

Agent 保持不变，但边界更清晰：

* 仍然是唯一语义层
* 仍然负责意图拆解与组合
* 通过 contract-aware schema 调用能力

---

# 六、契约系统设计（未来核心）

```json
{
  "node": "tag_rank",
  "inputs": {
    "token": "string",
    "aliases": ["string"]
  },
  "outputs": {

        "count": "int"
      }

    "stateless": true,
    "deterministic": true
  }
}
```

# 七、升级收益（未来状态）

## 7.1 插件完全独立

* 可跨语言实现
* 可独立部署
* 可独立扩展

---

## 7.2 系统解耦

* Gateway 不再依赖插件代码
* 插件不再依赖核心库

---

## 7.3 强类型保证（运行时）

* schema validation
* contract enforcement
* execution safety

---

## 7.4 可扩展性提升

* plugin = service
* capability = network node

---

# 八、当前阶段保留策略（重要）

由于当前系统处于**单人使用 + 快速迭代阶段**：

### 当前原则：

* 保留继承式 Node 结构
* 保持 in-process plugin model
* 不引入 RPC 层
* 不引入 contract registry system

---

### 原因：

* 降低开发复杂度
* 保持迭代速度
* 避免过早分布式化

---

# 九、迁移触发条件（未来再评估）

当满足以下条件时可启动升级：

* 插件数量显著增加
* 需要跨语言实现插件
* 需要插件独立部署
* Gateway 成为性能瓶颈
* 需要多机器并行执行能力

---

# 十、结论

当前系统设计保持不变，但已预留完整演进路径：

> 从“继承式插件单体执行系统”逐步演进为“基于契约的分布式能力网络系统”。

该设计保证：

* 当前阶段保持极简实现
* 未来可无损升级架构模型
* 插件能力可独立演化而不破坏核心系统
