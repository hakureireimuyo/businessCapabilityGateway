# 系统设计——节点协议与规范架构

> **状态**: 已实现（v2.0）。本文档记录系统设计思路，实际实现细节以代码和 Agent 使用手册为准。

---

## 一、设计动机

### 旧架构的根本问题

旧架构（v1）存在深层矛盾：

| 维度 | v1 设计 | 实际问题 |
|------|--------|---------|
| 节点模型 | Node + DAG + Artifact | 实际按 Source → Transform → Sink 三段式分类 |
| 执行方式 | 依赖驱动的图执行 | 线性管道迭代 |
| 组合方式 | 类型兼容性自动构图 | Agent 手动排列 BCL 文本顺序 |
| 验证方式 | 图合法性 | 管道合法性（首 Source、尾 Sink） |

核心问题：**NodeType（Source/Transform/Sink）把节点按"在管道中的位置"分类，这与 DAG 的本质矛盾。** 在真正的 DAG 中，有 `input_specs` 的节点如果这次不连任何上游，它就是事实上的入口。

### 新架构的核心转变

```
旧思维：节点是什么类型 → 节点能放在管道的什么位置 → 怎么按顺序执行
新思维：节点有什么规范 → 节点之间是否兼容     → 怎么按依赖执行
```

---

## 二、系统定位

**Business Capability Gateway 是一个面向 Agent 的节点协议平台。** 插件将能力封装为 Node，Agent 通过组合 Node 构建 Graph，Gateway 校验规范、解析依赖、按拓扑顺序调度执行。

### 核心关系

```
Agent（外部）
  │
  ├─ 查询: GET /plugins/<name>/nodes       → 节点概要列表
  │        GET /plugins/<name>/nodes/<id>  → 单个节点完整 Spec
  │
  ├─ 组合: 根据 Spec 的类型声明，用 Python SDK 构造 Graph
  │
  ├─ 提交: POST /execute (text/plain)
  │         body 为 Python 脚本
  │
  ▼
Gateway
  ├─ Sandbox: AST 校验 → restricted __builtins__ → exec
  ├─ GraphValidator: 节点存在 → 参数合法 → 输入完备 → 类型兼容 → 无环 → 输出合法
  ├─ GraphExecutor: 拓扑排序 → 依赖驱动的顺序调度
  │
  ▼
结构化结果 → Agent 自行推理
```

### 边界原则

- 系统只产出结构化数据和分析指标，不做 AI 推理
- 数据绝不跨插件领域混合
- 一次执行限定在单一插件内
- 跨领域综合分析由 Agent 在外部完成

## 三、核心概念

### 3.1 Node — 协议的基本实现单元

Node 不再分类型。每个 Node 只声明三件事：**需要什么输入** (`input_specs`)、**产出什么** (`output_spec`)、**接受什么参数** (`parameter_specs`)。

参考实现：`plugins/amazon/nodes/` 下各文件。

### 3.2 Artifact — 节点间的数据协议

- **ArtifactType**（编译时）：类型层次，子类型可流入期望父类型的输入。参考 `plugins/amazon/artifact_types.py`。
- **Artifact**（运行时）：带 key/type/data/produced_by/metadata 的数据载体。

类型兼容规则：`issubclass(上游产出类型, 下游期望类型)`。

### 3.3 Graph — 组合协议

Graph = 节点实例 + Artifact 连接边 + 输出声明。参考 `core/graph/model.py`。

Agent 通过 Python SDK 构建 Graph，不直接拼 JSON。参考 `gateway.py`。

### 3.4 ExecutionContext — 执行上下文

纯数据载体，不含管道字段。参考 `core/runtime/context.py`。

## 四、协议分层

```
Layer 4: 组合协议 — Graph 构建、边连接、输出声明       ← Agent 视角
Layer 3: 类型协议 — ArtifactType 层次、类型兼容         ← 插件开发者定义
Layer 2: 能力协议 — Node Spec 的定义与暴露              ← 插件开发者声明
Layer 1: 执行协议 — 依赖解析、顺序调度                  ← 网关负责
```

### Layer 1: 执行协议

依赖驱动的顺序调度。参考 `core/graph/executor.py`。

- 节点所有必填输入就绪 → Ready
- 所有 Ready 节点按拓扑顺序依次执行
- 图必须无环（校验阶段拦截）
- 节点失败 → 上游报错，不静默丢弃

### Layer 2: 能力协议

**两步发现 API**：

```
GET /plugins/<name>/nodes        → 概要列表（name, description, is_entry, input_count, output_key, output_type）
GET /plugins/<name>/nodes/<id>   → 单个节点完整 Spec（input_specs, output_spec, parameter_specs 全部细节）
```

Agent 先扫概要、挑候选节点、再请求完整 Spec。参考 `api/routes.py` 和 `core/registry/node_registry.py`。

### Layer 3: 类型协议

插件定义 ArtifactType 层次。参考 `plugins/amazon/artifact_types.py`。

类型校验发生在 Edge 建立时（GraphValidator Layer 4），不在执行时。

### Layer 4: 组合协议

Agent 通过 Python SDK 构建 Graph。参考 `gateway.py`。

图合法性规则（取代旧的管道规则）：

| 规则 | 说明 |
|------|------|
| 无环 | 不允许循环依赖 |
| 类型兼容 | 每条边 output_type 是 input_type 的子类 |
| 必填输入满足 | required=True 的 input 必须有入边 |
| 无悬空引用 | outputs 引用的 node_id 必须存在 |
| 参数有效 | params 匹配 parameter_specs |

## 五、Agent 接口

### HTTP 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/health` | 健康检查 |
| GET | `/plugins` | 已加载插件列表 |
| GET | `/plugins/<name>/nodes` | 节点概要列表 |
| GET | `/plugins/<name>/nodes/<id>` | 单个节点完整 Spec |
| POST | `/execute` | 执行 Graph（text/plain，Python 脚本） |

### SDK

Agent 使用 `Graph` 类构建图，参考 `gateway.py`：

- 创建：`g = Graph(plugin="amazon")`
- 添加节点：`products = g.keyword_search(keyword="test")`，`analysis = g.market_analysis(products=products)`
- 标记输出：`g.output(analysis)`
- 执行：`result = g.execute()`

所有方法名对应节点 `name`，字面量参数直接写值，Artifact 引用传占位符变量。

### 返回格式

```json
// 成功
{"success": true, "data": {"market_analysis": {...}}, "message": "Graph executed"}

// 失败
{"success": false, "error": {"code": "INVALID_GRAPH", "message": "...", "errors": [...]}}
```

## 六、GraphValidator — 6 层校验

参考 `core/graph/validator.py`：

| 层 | 检查内容 |
|----|---------|
| 1 | 节点存在性 — node_name 是否已注册 |
| 2 | 参数合法性 — params 匹配 parameter_specs |
| 3 | 输入完备性 — required=True 的 input 有入边 |
| 4 | 类型兼容 — issubclass(上游产出类型, 下游期望类型) |
| 5 | 无环 — Kahn 拓扑排序 |
| 6 | 输出合法性 — outputs 引用的 node_id 存在 |

---

## 七、Sandbox — 脚本安全执行

参考 `core/sandbox/`：

1. AST 校验：拦截 import、eval、exec、open、getattr、dunder 访问等
2. 受限 `__builtins__`：只暴露 Graph + 安全内置类型
3. 超时看门狗：10 秒上限，后台线程中断

## 八、插件架构

参考 `plugins/amazon/`：

```
plugins/<name>/
├── plugin.py              # register() 入口
├── artifact_types.py      # ArtifactType 层次
├── nodes/                 # Node 实现
├── services/              # 纯业务逻辑
└── repository/            # 数据访问
```

三层严格分离：Node → Service → Repository。

## 九、关键变更速查（v1 → v2）

| # | v1 | v2 |
|---|-----|-----|
| 1 | Node 分 Source/Transform/Sink | Node 不分类型，只声明协议 |
| 2 | BCL 文本 + DSL Parser | Python SDK 脚本 |
| 3 | PipelineExecutor 线性迭代 | GraphExecutor 拓扑排序 + 顺序执行 |
| 4 | 管道校验（首 Source 尾 Sink） | 图校验（无环 + 类型兼容） |
| 5 | ExecutionContext 含管道位置字段 | 纯数据载体 |
| 6 | execute(context) → context | execute(inputs, params, context) → Artifact |
| 7 | 能力发现返回完整 Spec | 两步发现：概要 → 详细 |
| 8 | 8 层 Linter | 6 层 GraphValidator |
| 9 | 无显式 Graph 数据结构 | Graph 为第一等公民 |
