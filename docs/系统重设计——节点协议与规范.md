# 系统重设计——节点协议与规范架构

> **状态：设计方案，尚未实现。**
> 本文档是整个系统的重新设计，核心变更：从"线性管道 + 节点分类"转变为"节点协议 + 规范驱动"。
> 旧有文档（架构设计与BCL语言确定结论.md、插件开发协议.md 等）中，凡与本文档冲突的内容，以本文档为准。

---

## 一、重设计动机

### 1.1 旧架构的根本问题

旧架构（即使是 DAG 版本）存在一个深层矛盾：**嘴上说图，身体走管道。**

| 维度 | 旧架构声称 | 旧架构实际 |
|------|-----------|-----------|
| 节点模型 | Node + DAG + Artifact | Source → Transform → Sink 三段式 |
| 执行方式 | 依赖驱动的并行图执行 | 按 BCL 行顺序的线性迭代 |
| 组合方式 | 类型兼容性自动构图 | Agent 在 BCL 中手动排列顺序 |
| 验证方式 | 图合法性（无环、类型匹配） | 管道合法性（首 Source、尾 Sink、中间 Transform） |

核心问题：**NodeType（Source/Transform/Sink）把节点按"在管道中的位置"分类，这与 DAG 的本质矛盾。** 在真正的 DAG 中，一个节点是"源"还是"转换"不由它的类别决定，而由它在这个特定图中的位置决定——一个有 `inputs` 的节点，如果这次调用不连任何上游，它就是事实上的"源"。

### 1.2 新架构的核心转变

```
旧思维：节点是什么类型 → 节点能放在管道的什么位置 → 怎么按顺序执行
新思维：节点有什么规范 → 节点之间是否兼容     → 怎么按依赖执行
```

**一句话：不再对节点分类，只对节点做规范约束。** 系统从一个"管道编排引擎"转变为一个"节点协议平台"。

## 二、系统新定位

### 2.1 一句话定位

**Business Capability Gateway 是一个面向 Agent 的节点协议平台。** 各业务插件将自身能力封装为符合节点协议的 Node，Agent 通过组合 Node 来描述分析任务，Gateway 负责校验规范兼容性、解析依赖关系、调度并行执行。

### 2.2 核心关系

```
Agent（外部）
  │
  ├─ 查询: "这个业务域有哪些能力节点？每个节点的规范是什么？"
  │         → GET /plugins/<name>/nodes 返回 Node Spec 列表
  │
  ├─ 组合: 根据规范兼容性，选择节点、声明连接关系、设定参数
  │         → 构造 Graph 对象（Python SDK 或 JSON 描述）
  │
  ├─ 提交: 将 Graph 描述提交给 Gateway
  │         → POST /execute
  │
  ▼
Gateway
  │
  ├─ 校验: 节点规范是否兼容？图是否无环？参数是否合法？
  ├─ 解析: 拓扑排序，识别并行组
  ├─ 调度: 依赖就绪即执行，无依赖的并行执行
  │
  ▼
结构化结果 → Agent 自行推理
```

### 2.3 边界原则（不变）

- 系统只产出结构化数据和分析指标，不做 AI 推理
- 数据绝不跨插件领域混合
- 一次执行严格限定在单一插件内
- 跨领域综合分析由 Agent 在外部完成

## 三、核心概念重新定义

### 3.1 Node（节点）—— 协议的基本实现单元

**Node 不再有类型。** 不再区分 Source / Transform / Sink。

每个 Node 是一个 **协议实现单元**，它只声明三件事：

```
我需要的输入  (input_specs)
我产出的输出  (output_spec)
我接受的参数  (parameter_specs)
```

Node 协议定义：

```python
class NodeSpec:
    """节点的完整规范——Agent 和 Gateway 之间的唯一契约"""
    name: str                          # 节点唯一标识（英文小写+下划线）
    plugin: str                        # 所属插件领域
    description: str                   # 业务说明（一句英文）

    # 输入规范：这个节点需要消费哪些 Artifact
    input_specs: dict[str, InputSpec]  # input_name → InputSpec

    # 输出规范：这个节点产出什么 Artifact
    output_spec: OutputSpec            # 单一输出（single output）

    # 参数规范：这个节点接受哪些字面量配置
    parameter_specs: dict[str, ParameterSpec]  # param_name → ParameterSpec


class InputSpec:
    name: str                  # 输入参数名
    artifact_type: type        # 期望的 Artifact 类型（issubclass 检查）
    required: bool = True      # 是否必须连接
    description: str = ""


class OutputSpec:
    key: str                   # 输出在 context.artifacts 中的存储 key
    artifact_type: type        # 产出的 Artifact 类型
    description: str = ""


class ParameterSpec:
    name: str
    param_type: type           # str | int | float | bool
    required: bool = False
    default: Any = None
    description: str = ""
```

**关键变更：**
- 没有 `node_type` 字段
- `input_specs` 是 dict（支持多个不同命名的输入）
- `output_spec` 是单个对象（每个 Node 只有单一输出）
- 所有声明都是强类型的（ArtifactType，不是 dict）

### 3.2 Artifact（产物）—— 节点间的数据协议

Artifact 是节点之间传递的数据产物的抽象。它有两个层面：

**层面一：Artifact 类型（编译时）**

```python
class ArtifactType:
    """所有 Artifact 类型的基类"""
    pass

# 类型层次由插件定义，例如 Amazon 插件：
# ArtifactType
#   ├── ProductCollection
#   │     └── FilteredProductCollection
#   ├── SalesMetrics
#   ├── CompetitionMetrics
#   ├── MarketAnalysis
#   ├── ChartData
#   └── ReportData
```

**层面二：Artifact 实例（运行时）**

```python
class Artifact:
    """运行时 Artifact 实例"""
    key: str             # 在 artifacts 字典中的唯一 key
    type: type           # ArtifactType 子类
    data: Any            # 实际数据
    produced_by: str     # 产出此 Artifact 的 node name
    metadata: dict       # {row_count, elapsed_ms, ...}
```

**类型兼容规则：**

```python
def is_compatible(output_type: type, input_spec: InputSpec) -> bool:
    """输出类型是否能满足输入规范"""
    return issubclass(output_type, input_spec.artifact_type)
```

子类型可以赋值给父类型。例如 `FilteredProductCollection` 可以流入一个期望 `ProductCollection` 的输入。

### 3.3 Graph（图）—— 组合协议

Graph 是 Node 实例 + 它们之间的 Artifact 连接构成的有向无环图。

```python
class Graph:
    plugin: str                              # 插件领域
    nodes: dict[str, GraphNode]               # node_id → GraphNode
    edges: list[GraphEdge]                    # 连接边
    outputs: list[str]                        # 哪些 node_id 的输出是最终产物

class GraphNode:
    node_id: str                # 此实例在图中的唯一 ID（如 "search_1"）
    node_name: str              # 对应的 Node 名称（如 "keyword_search"）
    params: dict[str, Any]      # 绑定的字面量参数
    # 注意：inputs 通过 edges 表达，不在此处声明

class GraphEdge:
    from_node: str              # 上游 node_id
    from_output: str            # 上游的 output key
    to_node: str                # 下游 node_id
    to_input: str               # 下游的 input name
```

**DAG 示例：**

```
Graph:
  nodes:
    search_1:  {node_name: "keyword_search",  params: {keyword: "halloween"}}
    sales_1:   {node_name: "sales_analysis",   params: {}}
    review_1:  {node_name: "review_analysis",  params: {}}
    market_1:  {node_name: "market_score",     params: {}}
    chart_1:   {node_name: "chart_output",     params: {}}
    report_1:  {node_name: "report_output",    params: {}}

  edges:
    search_1.products   → sales_1.products
    search_1.products   → review_1.products
    sales_1.sales       → market_1.sales
    review_1.reviews    → market_1.reviews
    market_1.market     → chart_1.market
    market_1.market     → report_1.market

  outputs: [chart_1, report_1]
```

### 3.4 ExecutionContext（执行上下文）—— 重新设计

去除管道相关字段，ExecutionContext 变为纯数据载体：

```python
class ExecutionContext:
    request_id: str                    # 请求唯一标识
    plugin_name: str                   # 当前插件名
    artifacts: dict[str, Artifact]     # 所有已产出的 Artifact（key → Artifact）
    outputs: dict[str, Any]            # 标记为输出的最终结果（output_key → data）
    metadata: dict[str, Any]           # 运行元信息（timestamps, counts）
```

**删除的字段：**
- ~~`pipeline_nodes`~~ → 被 Graph 取代
- ~~`current_node_index`~~ → 无意义（图不是线性）
- ~~`current_params`~~ → 传入 `execute(artifact_inputs, params)` 即可
- ~~`data`~~ → 改为 `artifacts` 字典，支持多 Artifact
- ~~`result`~~ → 改为 `outputs` 字典

**Node.execute 签名变更：**

```python
# 旧签名
def execute(self, context: ExecutionContext) -> ExecutionContext:
    keyword = context.current_params["keyword"]
    products = context.data
    context.data = result
    return context

# 新签名
def execute(
    self,
    inputs: dict[str, Artifact],   # 已解析的输入 Artifact
    params: dict[str, Any],        # 当前节点的字面量参数
    context: ExecutionContext,     # 全局上下文（只读 metadata，写入 artifacts/outputs）
) -> Artifact:                     # 返回产出的 Artifact
    keyword = params["keyword"]
    products = inputs["products"].data
    result_data = self._analyze(products)
    return Artifact(
        key=self.output_spec.key,
        type=self.output_spec.artifact_type,
        data=result_data,
        produced_by=self.name,
    )
```

**好处：**
- 输入/输出显式化，不需要从 context 隐式读取
- 无副作用（不修改 context 的中间字段）
- 类型安全（输入 Artifact 的类型在执行前已校验）

## 四、协议分层架构

整个系统可以理解为四层协议栈：

```
┌──────────────────────────────────────────────┐
│  Layer 4: 组合协议 (Composition Protocol)     │
│  图构建、边连接、输出声明                       │
│  ─────────────────────────────────────────   │
│  Layer 3: 类型协议 (Type Protocol)            │
│  ArtifactType 层次、类型兼容检查、子类型关系     │
│  ─────────────────────────────────────────   │
│  Layer 2: 能力协议 (Capability Protocol)      │
│  Node Spec 的定义、发现与匹配                   │
│  ─────────────────────────────────────────   │
│  Layer 1: 执行协议 (Execution Protocol)       │
│  Node 生命周期、依赖解析、并行调度、错误处理     │
└──────────────────────────────────────────────┘
```

### 4.1 Layer 1: 执行协议

**Node 生命周期：**

```
         register()
             │
             ▼
    ┌──────────────┐
    │  Registered   │   ← Node 已注册，可通过 /nodes 发现
    └──────┬───────┘
           │ graph 中包含此 Node
           ▼
    ┌──────────────┐
    │   Pending     │   ← 等待所有 input Artifact 就绪
    └──────┬───────┘
           │ 所有 input 就绪
           ▼
    ┌──────────────┐
    │   Running     │   ← execute(inputs, params, context)
    └──────┬───────┘
           │ 成功 / 失败
           ▼
    ┌──────────────┐
    │   Completed   │
    └──────────────┘
```

**执行规则：**

1. 节点所有必填 `input_specs` 都有对应的 Artifact 已就绪时，节点进入 Ready 状态
2. 所有 Ready 节点可并行调度执行
3. 图必须无环——在构建阶段检测并拒绝
4. 节点执行失败时，依赖它的下游节点标记为 Skipped
5. 当所有 `outputs` 中声明的节点都已完成（或已 Skip），图执行结束

**GraphExecutor 伪代码：**

```python
class GraphExecutor:
    def execute(self, graph: Graph) -> dict[str, Any]:
        context = ExecutionContext(plugin_name=graph.plugin)
        completed: set[str] = set()
        artifacts: dict[str, Artifact] = {}
        errors: dict[str, Exception] = {}

        while not all_outputs_ready(graph.outputs, completed, errors):
            # 找所有输入就绪且未执行的节点
            ready = find_ready_nodes(graph, artifacts, completed, errors)
            if not ready and not all_outputs_ready(...):
                raise DeadlockError("图中存在无法满足的依赖")

            # 并行执行所有就绪节点
            results = parallel_execute(ready, context)

            # 收集结果
            for node_id, result in results.items():
                if isinstance(result, Exception):
                    errors[node_id] = result
                else:
                    artifacts[result.key] = result
                completed.add(node_id)

        return {
            key: artifacts[key].data
            for key in graph.outputs
        }
```

### 4.2 Layer 2: 能力协议

**能力发现 API：**

```
GET /plugins/<name>/nodes

返回：
{
  "plugin": "amazon",
  "nodes": [
    {
      "name": "keyword_search",
      "description": "Search products by keyword",
      "input_specs": {},
      "output_spec": {
        "key": "products",
        "artifact_type": "ProductCollection"
      },
      "parameter_specs": {
        "keyword": {"type": "str", "required": true, "description": "Search keyword"},
        "limit":   {"type": "int", "required": false, "default": 50}
      }
    },
    {
      "name": "market_analysis",
      "description": "Analyze market competition",
      "input_specs": {
        "products": {
          "artifact_type": "ProductCollection",
          "required": true,
          "description": "Products to analyze"
        }
      },
      "output_spec": {
        "key": "market_analysis",
        "artifact_type": "MarketAnalysis"
      },
      "parameter_specs": {}
    }
  ]
}
```

**关键：** Node Spec 是对 Agent 的承诺——Agent 完全依赖 Spec 来规划图的组合。Spec 必须准确、完整、自描述。

### 4.3 Layer 3: 类型协议

**类型层次：**

```
ArtifactType                    # 抽象基类
  │
  ├── RawData                   # 原始数据标记（mixin）
  │   └── ProductCollection
  │       └── FilteredProductCollection
  │
  ├── Metric                    # 分析指标标记（mixin）
  │   ├── SalesMetrics
  │   ├── ReviewMetrics
  │   ├── CompetitionAnalysis
  │   ├── MarketAnalysis
  │   └── OpportunityList
  │
  └── Aggregation               # 聚合结果标记（mixin）
      ├── ChartData
      ├── ReportData
      └── JSONData
```

**兼容性矩阵：**

| 输出类型 | 输入期望 | 兼容？ |
|---------|---------|--------|
| `FilteredProductCollection` | `ProductCollection` | ✅ 子类型 |
| `ProductCollection` | `FilteredProductCollection` | ❌ 父类型不兼容子类型 |
| `SalesMetrics` | `ProductCollection` | ❌ 类型不相关 |
| `MarketAnalysis` | `MarketAnalysis` | ✅ 同类型 |

**类型校验位置：** 在 Edge 建立时校验，而非执行时。

```python
def validate_edge(edge: GraphEdge, from_node_spec: NodeSpec, to_node_spec: NodeSpec):
    output_type = from_node_spec.output_spec.artifact_type
    input_spec = to_node_spec.input_specs[edge.to_input]
    if not issubclass(output_type, input_spec.artifact_type):
        raise TypeMismatchError(
            f"Edge {edge.from_node}.{edge.from_output} → {edge.to_node}.{edge.to_input}: "
            f"type {output_type.__name__} is not compatible with expected {input_spec.artifact_type.__name__}"
        )
```

### 4.4 Layer 4: 组合协议

**图的合法性规则（取代旧的管道规则）：**

| 规则 | 说明 |
|------|------|
| **无环** | 图中不允许存在循环依赖 |
| **类型兼容** | 每条边的输出类型必须与输入期望兼容 |
| **必填输入满足** | 每个节点的所有 `required=True` 的 input_specs 必须有入边 |
| **无悬空引用** | 所有 `outputs` 引用的 node_id 必须存在于图中 |
| **单插件约束** | 图中所有节点必须属于同一插件 |
| **参数有效** | 每个节点的 params 必须匹配其 parameter_specs |

**不再存在的规则：**
- ~~第一个节点必须是 Source~~
- ~~最后一个节点必须是 Sink~~
- ~~中间节点必须是 Transform~~
- ~~Sink 后面不能跟节点~~

### 4.5 协议依赖图

```
┌────────────────────────────────────────────┐
│                                            │
│   能力协议 ──→ 类型协议 ──→ 组合协议 ──→ 执行协议  │
│      │             │             │           │
│   Node Spec    ArtifactType   Graph        │
│   的定义与      的层次与       的构建与      GraphExecutor
│   发现         兼容检查       校验          │
│                                            │
└────────────────────────────────────────────┘
```

上层依赖下层，下层不感知上层：
- 能力协议使用类型协议来声明输入/输出类型
- 组合协议使用类型协议来校验边的兼容性
- 执行协议执行组合协议产出的 Graph

## 五、Agent 接口层

### 5.1 Python SDK（主接口）

```python
from gateway import Graph

g = Graph(plugin="amazon")

# 每个方法调用 = 声明一个节点实例
# 返回 ArtifactPlaceholder（不是真实数据）
products = g.keyword_search(keyword="halloween garland")

# 分支：两个节点都消费 products，互不依赖，可并行
sales = g.sales_analysis(products=products)
reviews = g.review_analysis(products=products)

# 汇合：market_score 消费两个 Artifact
market = g.market_score(sales=sales, reviews=reviews)

# 多输出
g.output(g.chart_output(market=market))
g.output(g.report_output(market=market))

# 执行
result = g.execute()
# result = {"chart_output": ChartData(...), "report_output": ReportData(...)}
```

**API 规则：**

| 规则 | 说明 |
|------|------|
| 全部命名参数 | 禁止位置参数 |
| 字面量参数直接写值 | `keyword="halloween"`, `limit=50` |
| Artifact 引用直接传变量 | `products=products` |
| 必须有赋值目标 | `x = g.some_node(...)` |
| 显式标记输出 | `g.output(...)` |
| 显式执行 | `g.execute()` 才真正运行 |

### 5.2 禁止的控制流（不变）

```python
# ❌ 条件
if something:
    g.some_node(...)

# ❌ 循环
for x in items:
    g.some_node(...)

# ❌ 列表推导
[g.some_node(x) for x in items]
```

条件过滤通过节点参数（AND）和图结构（OR）表达，与旧设计一致。

### 5.3 HTTP 接口

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/health` | 健康检查 |
| GET | `/plugins` | 已加载插件列表 |
| GET | `/plugins/<name>/nodes` | 某插件的所有节点 Spec |
| POST | `/execute` | 执行 Graph（body 为 Python 脚本或 JSON Graph 描述） |

**JSON Graph 描述格式（作为 HTTP body 的备选）：**

```json
{
  "plugin": "amazon",
  "nodes": {
    "s1": {"node_name": "keyword_search", "params": {"keyword": "halloween"}},
    "a1": {"node_name": "sales_analysis", "params": {}},
    "a2": {"node_name": "review_analysis", "params": {}},
    "m1": {"node_name": "market_score", "params": {}},
    "o1": {"node_name": "chart_output", "params": {}},
    "o2": {"node_name": "report_output", "params": {}}
  },
  "edges": [
    {"from": "s1", "from_output": "products",  "to": "a1", "to_input": "products"},
    {"from": "s1", "from_output": "products",  "to": "a2", "to_input": "products"},
    {"from": "a1", "from_output": "sales",     "to": "m1", "to_input": "sales"},
    {"from": "a2", "from_output": "reviews",   "to": "m1", "to_input": "reviews"},
    {"from": "m1", "from_output": "market",    "to": "o1", "to_input": "market"},
    {"from": "m1", "from_output": "market",    "to": "o2", "to_input": "market"}
  ],
  "outputs": ["o1", "o2"]
}
```

### 5.4 统一返回格式

```json
// 成功
{
  "success": true,
  "data": {
    "chart_output": { "market_size": 523, "avg_price": 18.7 },
    "report_output": { "markdown": "...", "summary": "..." }
  },
  "message": "Graph executed (3 nodes, 2 parallel, 42ms)"
}

// 失败
{
  "success": false,
  "error": {
    "code": "TYPE_MISMATCH",
    "message": "Edge s1.products → a1.products: ProductCollection is not compatible with expected SalesData"
  }
}
```

**错误码体系：**

| 错误码 | 含义 |
|--------|------|
| `TYPE_MISMATCH` | Artifact 类型不兼容 |
| `NODE_NOT_FOUND` | 引用的 node_name 未注册 |
| `PLUGIN_NOT_FOUND` | 插件不存在 |
| `INVALID_PARAMS` | 参数类型或必填校验失败 |
| `CYCLIC_DEPENDENCY` | 图中存在循环依赖 |
| `UNSATISFIED_INPUT` | 某个节点的必填输入没有入边 |
| `DANGLING_OUTPUT` | output 引用了不存在的 node_id |
| `CROSS_PLUGIN` | 跨插件引用 |
| `EXECUTION_FAILED` | 节点执行异常 |
| `DEADLOCK` | 存在无法满足的依赖链 |

---

## 六、规范校验器（Validator）

取代旧的 Linter（BCL 语法→符号→节点→参数→类型→图→业务→输出），新的 Validator 面向 Graph 对象，校验层次为：

### 校验层次

| 层 | 检查内容 | 示例错误 |
|----|---------|---------|
| 1. 节点存在性 | graph 中引用的每个 node_name 是否在插件中注册 | `"sales_analysis"` 不在 amazon 插件中 |
| 2. 参数合法性 | 每个节点的 params 是否匹配其 parameter_specs | `keyword=123`，期望 `str` |
| 3. 输入完备性 | 每个节点的必填 input 是否都有入边 | `market_score.sales` 没有入边 |
| 4. Artifact 类型兼容 | 每条边的输出类型是否满足输入期望 | `ProductCollection` → 期望 `SalesMetrics` |
| 5. 图无环 | 是否存在循环依赖 | s1 → a1 → m1 → s1 |
| 6. 输出合法性 | outputs 引用的 node_id 是否存在 | `outputs: ["ghost_node"]` |
| 7. 业务规则 | 单插件约束、无跨插件边 | 同时引用 amazon 和 tiktok 节点 |

### 输出格式

```json
{
  "valid": false,
  "errors": [
    {
      "layer": "TYPE_MISMATCH",
      "edge": {"from": "s1", "to": "a1"},
      "expected": "SalesMetrics",
      "actual": "ProductCollection",
      "message": "Type mismatch: edge s1.products → a1.sales expects SalesMetrics, got ProductCollection"
    }
  ]
}
```

## 八、Node 实现示例（新协议）

### 8.1 无输入节点（原 "Source"）

```python
class KeywordSearchNode(Node):
    name = "keyword_search"
    plugin = "amazon"
    description = "Search products by keyword"

    input_specs = {}   # 无输入

    output_spec = OutputSpec(
        key="products",
        artifact_type=ProductCollection,
        description="Matching products",
    )

    parameter_specs = {
        "keyword": ParameterSpec("keyword", str, required=True, description="Search keyword"),
        "limit": ParameterSpec("limit", int, required=False, default=50, description="Max results"),
    }

    def execute(
        self,
        inputs: dict[str, Artifact],
        params: dict[str, Any],
        context: ExecutionContext,
    ) -> Artifact:
        keyword = params["keyword"]
        limit = params.get("limit", 50)
        products = self.repository.search(keyword, limit)

        return Artifact(
            key=self.output_spec.key,
            type=ProductCollection,
            data=products,
            produced_by=self.name,
            metadata={"count": len(products)},
        )
```

class MarketScoreNode(Node):
    name = "market_score"
    plugin = "amazon"
    description = "Compute aggregate market score from multiple metrics"

    input_specs = {
        "sales": InputSpec(
            name="sales",
            artifact_type=SalesMetrics,
            required=True,
            description="Sales metrics input",
        ),
        "reviews": InputSpec(
            name="reviews",

            required=True,

        ),

    output_spec = OutputSpec(

        artifact_type=MarketSignal,
        description="Aggregated market signal",

        review_data = inputs["reviews"].data

            key=self.output_spec.key,
            type=MarketSignal,
            data=signal,

## 九、执行示例对比

### 旧架构（管道思维）

```
BCL:
  amazon/keyword_search&keyword=halloween
  > sales_analysis()
  > market_score()
  > chart_output()

执行:
  for node in pipeline:
      context = node.execute(context)
  return context.result
```

### 新架构（协议+图思维）

```
Graph (JSON 或 SDK 构造):
  nodes: {search, sales, reviews, market, chart, report}
  edges: [
    search→sales, search→reviews,
    sales→market, reviews→market,
    market→chart, market→report
  ]
  outputs: [chart, report]

执行:
  ready = [search]           # 无依赖 → 立即执行
  search 完成 → ready = [sales, reviews]  # sales 和 reviews 并行执行
  sales+reviews 完成 → ready = [market]
  market 完成 → ready = [chart, report]  # chart 和 report 并行执行
  所有 outputs 完成 → 返回结果
```

## 十一、关键变更速查

| # | 旧设计 | 新设计 |
|---|--------|--------|
| 1 | Node 分三类型：Source / Transform / Sink | Node 不分类型，只声明 input_specs + output_spec |
| 2 | 管道校验：首 Source 尾 Sink | 图校验：无环 + 类型兼容 + 输入满足 |
| 3 | PipelineExecutor 顺序迭代 | GraphExecutor 依赖驱动并行调度 |
| 4 | ExecutionContext 含 pipeline_nodes / current_node_index / data | 去除管道字段，artifacts 为 dict |
| 5 | Node.execute(context) → context | Node.execute(inputs, params, context) → Artifact |
| 6 | input_schema / output_schema 为 dict | input_specs / output_spec 为强类型 Spec 对象 |
| 7 | 参数校验在 Node.validate_params | 参数校验纳入 GraphValidator |
| 8 | BCL 文本 + DSL Parser | Python SDK (主) + JSON Graph 描述 (辅) |
| 9 | 能力发现返回 "type": "source/transform/sink" | 能力发现返回 input_specs / output_spec |
| 10 | Linter 八层检查 | Validator 七层检查 |
| 11 | 无显式 Graph 数据结构 | Graph + GraphNode + GraphEdge 为第一等公民 |
| 12 | 节点间通过 context.data 隐式传数据 | 节点间通过 Artifact 显式传数据 |

## 十二、设计原则总结

```
┌─────────────────────────────────────────────────┐
│                                                 │
│  协议定义契约  ──  Node Spec 是唯一的接口约定       │
│  规范驱动组合  ──  类型兼容性决定边是否合法         │
│  依赖驱动执行  ──  就绪即执行，不按声明顺序         │
│                                                 │
│  Node 不分类型  ──  节点只有规范约束，没有分类标签   │
│  Graph 是第一等公民 ──  不只存在于注释中            │
│  Artifact 是显式协议 ──  不是隐式的 context.data    │
│                                                 │
│  系统不做推理 ──  只产出结构化数据                  │
│  数据不跨领域 ──  一次图执行限定单一插件             │
│  插件不改网关 ──  Open for Extension               │
│                                                 │
└─────────────────────────────────────────────────┘
```
