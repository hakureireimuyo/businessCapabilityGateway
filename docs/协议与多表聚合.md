# 我建议的修改

如果考虑未来扩展，我会改成：

```python

    request_id: str

    datasets: dict[str, Any]

    result: dict

    metadata: dict
```

而不是：

```python

```

---

例如：

```python
datasets = {
    "products": ProductCollection,
    "sales": SalesCollection,
    "keywords": KeywordCollection
}
```

或者：

```python
datasets = {
    "halloween": ProductCollection,
    "christmas": ProductCollection
}
```

这样：

- 多表聚合
- 多数据集比较
- 未来 DAG

都能兼容。

---

因此结论是：

1. **当前协议可以支持绝大多数单业务领域分析。**
2. **多表 Join 不需要修改协议，只需要让 Source 构造领域聚合对象，而不是返回单表对象。**
3. **跨多个独立数据集的比较分析，目前协议无法优雅表达。**
4. **`ExecutionContext.data` 是未来最大的扩展瓶颈，建议尽早升级成 `datasets`。**
5. **如果未来出现多个 Source 汇聚，则线性 Pipeline 会升级成 DAG，此时 DSL 和执行器都需要重新设计。**

对于你现在的 Amazon 场景，我认为还远没到需要 DAG 的程度，先把 **"表 → 领域对象 → Collection"** 这一层抽象做好，比引入多 Source 更重要。



而不是：


# 文档二：BCL语言设计规范

**Document Title: **

# Business Composition Language (BCL)

## 设计目标

BCL 是业务编排语言。

BCL 负责描述：

- 调用哪些能力
- 能力之间的依赖关系
- 最终输出什么

BCL 不负责：

- 执行
- 调度
- 并发控制

这些由 Runtime 完成。

---

# 设计原则

## 原则一

一种概念只允许一种语法。

---

## 原则二

所有节点调用形式一致。

---

## 原则三

所有依赖通过变量引用表达。

---

## 原则四

不支持流程控制语句。

禁止：

if

while

for

switch

lambda

---

BCL 不是通用编程语言。

BCL 是业务编排语言。

# 目前我认为最大的架构风险

实际上不是语法。

不是图。

不是类型系统。

而是：

## 你还没有定义 Artifact

现在讨论中：

```text
ProductCollection

SalesMetrics

MarketSignal
```

都被称为 Artifact。

---

但实际上：

```text
Artifact
```

才是整个系统真正的核心。

---

因为：

```text
Node
```

只是：

```text
Artifact
→
Artifact
```

的转换器。

---

Graph也只是：

```text
Artifact
之间的依赖图
```

---

所以我下一步会优先写：

```text
Artifact Type System Design
```

而不是继续扩展BCL。

至少先明确：

```python
Artifact

RawDataArtifact

MetricArtifact

AggregationArtifact
```

---

以及：

```python
ArtifactSchema

ArtifactCategory

ArtifactStorage

ArtifactReference
```

---

因为从你目前的定位来看：

```text
Agent
↓
BCL
↓
Graph
↓
Node
↓
Artifact
↓
Agent
```

整个系统最终传递的其实不是节点，不是图，而是 **Artifact**。

如果 Artifact 设计得好，Node 和 Graph 都会变得简单；如果 Artifact 设计得模糊，后面所有类型检查、跨业务聚合、缓存、复用、增量计算都会越来越难处理。

所以我认为目前剩下的最大架构空白就是：

```text
Artifact System
```

它甚至比 BCL 本身更值得优先设计。
