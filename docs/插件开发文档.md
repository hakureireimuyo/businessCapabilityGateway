# 插件开发文档

> **版本**: v2.0
> **面向读者**: 插件开发者
> **参考实现**: `plugins/amazon/` — 打开这个目录对照着看

---

## 一、核心概念

**Node 不分类型。** 一个节点是"入口"还是"转换"还是"终点"，不由标签决定，而由它在这个特定图中的位置决定。有 `input_specs` 的节点如果这次不连任何上游，它就是事实上的入口。

每个 Node 只声明三样东西：

| 声明 | 位置 | 含义 |
|------|------|------|
| `input_specs` | `plugins/amazon/nodes/` 各文件 | 我需要消费哪些 Artifact，每个的类型、是否必填 |
| `output_spec` | 同上 | 我产出什么 Artifact，键名和类型 |
| `parameter_specs` | 同上 | 我接受哪些字面量参数，类型、默认值、是否必填 |

## 二、目录结构

参考 `plugins/amazon/`：

```
plugins/<plugin_name>/
├── plugin.py              # 入口：register() 函数
├── artifact_types.py      # ArtifactType 子类层次
├── nodes/                 # Node 实现（目录按职责分，约定而已，网关不关心）
│   ├── __init__.py
│   ├── source_nodes.py    # 入口节点（无 input_specs）
│   ├── transform_nodes.py # 转换节点
│   └── sink_nodes.py      # 分析和输出节点
├── services/              # 纯业务逻辑
│   ├── __init__.py
│   └── market_service.py
└── repository/            # 数据访问
    ├── __init__.py
    └── product_repository.py
```

目录只是约定，网关不强制。你也可以把所有 Node 写在 `nodes.py` 单文件中。

## 三、插件内部三层

```
Node 层 （协议实现 + 对外暴露）
  ├── 声明 input_specs / output_specs / parameter_specs
  ├── 实现 execute(inputs, params, context) → Artifact
  └── 拆包 Artifact → 调用 Service → 打包为 Artifact

Service 层 （纯业务逻辑）
  ├── 接收普通 Python 数据结构（不是 Artifact）
  ├── 执行统计、评分、算法
  └── 返回 dict / list

Repository 层 （数据访问）
  ├── 定义数据模型（@dataclass）
  ├── 定义数据容器（支持链式过滤的 Collection 类）
  └── 管理模拟数据或真实数据源
```

**严格分离**：Node → Service → Repository，每层只认识下一层，不认识上一层。Service 从不知道自己在被 Node 调用，Node 不知道数据从哪来。

具体例子看 `plugins/amazon/nodes/sink_nodes.py` 中任意节点 → 调用 `plugins/amazon/services/market_service.py` → 接收 `plugins/amazon/repository/product_repository.py` 的 `ProductCollection`。

## 四、开发步骤

### 步骤 1：定义 Artifact 类型层次

在 `artifact_types.py` 中继承 `ArtifactType` 定义业务类型。**看 `plugins/amazon/artifact_types.py`**。

关键规则：
- 子类型 = "是更具体的父类型"。`FilteredProductCollection` 继承 `ProductCollection` → 可流入任何期望 `ProductCollection` 的输入
- 不相关的类型互不兼容（`SalesMetrics` 不能流入期望 `ProductCollection` 的输入）
- 类型层次保持在 1-2 层
- 通用消费节点可以将输入类型设为 `ArtifactType`（基类），接受任何 Artifact

### 步骤 2：实现 Node

继承 `core.protocol.node.Node`，声明协议属性，实现 `execute()`。**每种模式看一个具体文件即可**：

| 模式 | 参考文件 | 特征 |
|------|---------|------|
| 图入口节点 | `plugins/amazon/nodes/source_nodes.py` | `input_specs = {}`，`__init__` 持有 Repository |
| 数据转换节点 | `plugins/amazon/nodes/transform_nodes.py` | 输入输出同一类型体系，产出子类型 |
| 分析节点 | `plugins/amazon/nodes/sink_nodes.py` 中 `MarketAnalysisNode` / `SalesAnalysisNode` | 输入数据 → 产出指标 |
| 多输入汇合 | `plugins/amazon/nodes/sink_nodes.py` 中 `MarketScoreNode` | 两个不同类型输入 → 一个输出 |
| 通用消费节点 | `plugins/amazon/nodes/sink_nodes.py` 中 `ChartOutputNode` / `JSONOutputNode` | 输入类型为基类 `ArtifactType` |

**Node 约束**：

| 约束 | 说明 |
|------|------|
| 无状态 | `execute()` 每次调用独立，共享一个 Node 实例是安全的 |
| 不调用其他 Node | Node 间的连接只通过 Graph 边表达 |
| 单一输出 | 只有一个 `output_spec`，多输出需求拆分 Node |
| 不修改 context | Node 不写 `context.artifacts`（网关负责） |

**命名规范**：全部英文小写+下划线：`keyword_search`、`market_analysis`、`price_gte`。

### 步骤 3：实现 Service 层

参考 `plugins/amazon/services/market_service.py`。纯静态方法，接收 `ProductCollection`（不是 `Artifact`），返回 `dict`，不导入 `core.protocol` 的任何东西。

### 步骤 4：实现 Repository 层

参考 `plugins/amazon/repository/product_repository.py`。三部分：
1. **数据模型**：`@dataclass` 定义的 `Product`
2. **数据容器**：`ProductCollection` 类，支持链式过滤（每个 filter 方法返回新实例）和聚合方法
3. **数据仓库**：`ProductRepository` 类，负责存储和查询，返回 `ProductCollection`

### 步骤 5：创建插件入口

参考 `plugins/amazon/plugin.py`。只需一个 `register()` 函数：

```python
from core.registry.node_registry import register_nodes

def register():
    register_nodes("plugin_name", [
        NodeA(),
        NodeB(),
        ...
    ])
```

**约束**：
- 函数名固定为 `register`，无参数，在模块顶层
- Node 实例化一次即可（无状态，可共享）
- 不需要修改 `core/` 下的任何代码
- 文件放在 `plugins/` 下一个新目录，网关启动时自动发现

## 五、Graph 校验规则

你的 Node Spec 声明会被网关校验。校验分 6 层（原 7 层中第 7 层与第 1 层重复，已合并）：

| 层 | 检查内容 | 你的哪个声明被检查 |
|----|---------|------------------|
| 1. 节点存在 | `node_name` 是否已注册 | `plugin` + `name` |
| 2. 参数合法 | 参数类型、必填、无多余参数 | `parameter_specs` |
| 3. 输入完备 | 每个 `required=True` 的输入是否有入边 | `input_specs` |
| 4. 类型兼容 | 上游产出类型是否是下游期望类型的子类 | `output_spec.artifact_type` vs `input_specs[name].artifact_type` |
| 5. 无环 | 图中是否存在循环 | 图结构 |
| 6. 输出合法 | `outputs` 引用的 node_id 是否存在 | 图结构 |

## 六、开发检查清单

- [ ] `artifact_types.py` 中所有类型继承自 `ArtifactType`
- [ ] 每个 Node 的 `name` 在插件内唯一
- [ ] `output_spec.key` 有业务含义，不是 `data`、`result` 这种泛名
- [ ] `input_specs` 的 `artifact_type` 层次合适（不过于具体，不过于宽泛）
- [ ] `execute()` 只通过 `inputs` 和 `params` 获取数据，不访问外部状态（除了 Repository）
- [ ] Service 方法用 `@staticmethod`，不 import `Artifact`
- [ ] 不修改 `core/` 下的任何文件

## 不要做的事

- 不要给 Node 分类型标签
- 不要在 Node 之间直接调用
- 不要在 Service 层接触 Artifact
- 不要跨插件引用其他插件的 Node 或 ArtifactType
